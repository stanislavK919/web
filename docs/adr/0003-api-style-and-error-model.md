### 1. Архітектурний стиль: REST API
Ми дотримуємося принципів REST (Level 2 - Resources + Verbs):
* **Ресурси:** Використовуємо іменники у множині для URL (наприклад, `/todos`, `/users`).
* **Дії:** Визначаються HTTP-методами:
    * `GET`: Отримання даних.
    * `POST`: Створення нових ресурсів.
    * `PUT/PATCH`: Оновлення ресурсів.
    * `DELETE`: Видалення.
* **Stateless:** Сервер не зберігає стан клієнта між запитами (окрім сесії/токена авторизації).

### 2. Формат помилок
У разі виникнення помилки (4xx або 5xx), API завжди повертає JSON наступної структури:

```json
{
  "error": "ValidationError",      // Тип помилки (readable code)
  "code": "TITLE_REQUIRED",        // Унікальний код для frontend-логіки
  "details": [                     // Масив деталей (опціонально)
    {
      "field": "title",            // Поле, де сталася помилка
      "message": "Title cannot be empty" // Повідомлення для користувача
    }
  ]
}
```

### 3. Політика HTTP статус-кодів
Ми використовуємо стандартні коди для семантичного опису результату:

| Код | Значення | Використання |
| :--- | :--- | :--- |
| **200** | OK | Успішний синхронний запит, що повертає дані. |
| **201** | Created | Ресурс успішно створено (POST). |
| **204** | No Content | Успішний запит без тіла відповіді (DELETE). |
| **400** | Bad Request | Помилка валідації вхідних даних (клієнт винен). |
| **401** | Unauthorized | Користувач не авторизований. |
| **403** | Forbidden | Користувач авторизований, але не має прав. |
| **404** | Not Found | Ресурс не знайдено. |
| **500** | Internal Error | Необроблена помилка на сервері. |

### 4. Вимога відповідності
* docs/api/openapi.yaml визначає канонічну структуру API системи.
* Код (контролери та DTO) повинен суворо відповідати специфікації.
* Будь-яка зміна в API спочатку вноситься в YAML, обговорюється, і лише потім імплементується в коді.
