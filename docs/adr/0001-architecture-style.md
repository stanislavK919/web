# 1. Вибір архітектурного стилю: Модульний Моноліт

**Date:** 2025-12-03
**Status:** Accepted

## Вибір
Ми обираємо **Модульний Моноліт (Modular Monolith)**.

Це означає, що весь код (API, бізнес-логіка, робота з БД) виконується в рамках одного процесу розгортання, але всередині коду підтримується сувора модульність (логічний поділ, а не фізичний).

## Критерії перегляду рішення
Перехід до мікросервісів (або виділення окремих сервісів) при настанні однієї з таких умов:
1.  **Масштабування команди:** Над проєктом працює більше 3-х команд, і вони блокують одна одну при мерджі коду.
2.  **Гетерогенне навантаження:** Окремий модуль (наприклад, генерація звітів) потребує в 100 разів більше ресурсів CPU/RAM, ніж решта системи, і його дешевше масштабувати окремо.
3.  **Технологічні обмеження:** Частину функціонала необхідно реалізувати на іншій мові програмування (наприклад, Python для ML, Go для HighLoad).
4.  **Час білда:** Час CI/CD циклу перевищує 15-20 хвилин.

## Наслідки
* **Positive:** Спрощений деплой (один артефакт), рефакторинг виконується засобами IDE (безпечно переміщувати класи/методи), відсутність розподілених транзакцій.
* **Negative:** Єдина точка відмови (якщо процес падає — падає вся система), ризик сильного зв'язування коду (Tight Coupling), якщо не дотримуватися правил шарів.
